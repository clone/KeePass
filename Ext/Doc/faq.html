How can I help you?

If you like KeePass and would like to help me in some way:
  • Donate
    This is maybe the best way of helping me, if you don't have that much time or experience in application development. This is the recommended way of helping me for "standard users".
  • Make a translation
    If you have some free time, you could make a translation of KeePass (of course only if you're language isn't offered already)!
  • Test new releases and submit bugfixes
    KeePass is under constant development, new features get implemented, bugs get fixed. If you have some free time, you could just test the new releases thoroughly and if you're a programmer, look through the sources to find bugs and maybe even submit fixes.
  • Spread the word
    KeePass isn't that widely-known yet. If you like KeePass, tell all your friends how great KeePass is, publish articles about it, press it on CDs/DVDs, sell it with your USB sticks, submit it to software archives, talk in forums about it, etc.!


Multi-user databases

Implementing a multi-user system for KeePass is a frequent feature request...

All these multi-user / view-only user things are only working if you're using a client-server system, where the user doesn't have full access to the underlying data structures.

For KeePass, a multi-user system would be secure only on the GUI level (since it isn't a client-server application). I could add flags or something like ACLs (access control lists) to prevent/allow users editing/viewing defined sections of a database, but this isn't secure. For normal users (i.e. not programmers and not security experts) the ACLs would work very fine, but anyone who has just a very few bit experience in security can remove those ACLs, if he can read the database.

KeePass is open-source, that means that everyone can get the source code, modify it and recompile it. If it would be only secure on the GUI level, an experienced programmer could just take the KeePass sources, remove all the ACL stuff and would have full access to the database, if he knows the main password.

Since KeePass isn't a client-server application, it's "all or nothing": either you can read and write 100% of the current database or you cannot do anything with it.

So, I will NOT implement those multi-user features. Of course some people would be happy about such features (quote: "I only need it to be secure on a GUI level!"), but soon someone would complain that anyone can modify entries of other users in multi-user databases and KeePass isn't secure. I won't implement such an insecure system.

Some desktop (i.e. not server-client) password manager programs claim to have a working group rights management system, but this just isn't true. They're using ACLs, fine, but this only works on the GUI level and because their product isn't open-source. If you're only USING the program, the multi-user system works. If someone reverse-engineers the file format, he can do anything with the database, just as described above.

Maybe you don't believe me what I'm saying here, since many programs offer multi-user features. In this case, please read the sites below about Excel cell protection security. Protecting certain parts of an Excel file is exactly the same problem as protecting a KeePass database from other users. Even Microsoft says that their cell protections just work for normal users and not on the security level.

Please don't confuse this with other multi-user applications now! Multi-user systems CAN work, but only if the access of a user can really be restricted, for example when you're communicating with some server over the internet. In such a case multi-user systems work perfectly and are also 100% secure. It just doesn't and cannot work with the KeePass desktop application.

Will there be KeePass versions for other OSs than Windows?

The desktop KeePass application has been developed from start on to work on Windows operating systems. It uses the Microsoft Foundation Classes (MFC) and several other platform-dependent stuff that makes porting relatively hard.

Anyway, the KeePass core (i.e. the password management code, simply everything except the graphical user interface [GUI]) is platform-independent and therefor "ports" are theoretically possible, although you cannot really speak of a port here, since the whole user interface needs to be rewritten.

Also, KeePass is written in C++. No chance to convert it to Java to make it platform-independent. If you would like a KeePass written in Java, have fun rewriting it completely from scratch. Although C++ and Java are relatively similar, converting the existing C++ code to Java involves much work, most probably more work than just writing a completely new password manager in Java.

But people can make KeePass "ports", as one can easily see: KeePass for PocketPC. I assume that, when rewriting the GUI code, you can "port" KeePass to almost any platform, since the core password manager code is totally platform-independent.

Now before you ask when the Linux/PalmOS/etc. versions are ready: I don't know. I myself just don't have the time to develop these versions. Keep in mind that I'm developing KeePass in my free time and, apart from donations, don't get anything for it. I will happily develop KeePass for Windows, also in the future yes, but I don't have the time to develop a Linux version for example. I also never programmed any single PalmOS program, I don't even own a PalmOS.

KeePass is Open-Source. If you have the time to develop KeePass-compatible applications (as said, it cannot be really "ported"), feel free to contact me and I will happily support you as good as I can.
